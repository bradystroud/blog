# Brady blog writing guide

Use this as the default style guide whenever AI rewrites a draft into a real post.

## Core voice

- Practical beats performative.
- Opinionated is good when grounded in real experience.
- Sound like an engineer who actually did the work, not a content marketer summarising trends.
- Keep it concise, but not thin.
- Use plain language over inflated language.

## What good Brady posts usually do

- Open with the real problem, frustration, or observation.
- Use concrete examples from real work.
- Explain the trade-offs, not just the final answer.
- Teach something useful someone can apply immediately.
- End with a clear takeaway or reflection.

## Things to avoid

- Generic AI filler like "In today's fast-paced world" or "revolutionary".
- Fake certainty when the evidence is limited.
- Empty summaries of commits without a narrative.
- Overly academic explanations when a practical example would do.
- Long intros that delay the actual point.

## Preferred structure

A good default blog structure is:

1. Hook with the real issue
2. Explain why it mattered
3. Walk through what changed
4. Highlight the interesting lesson / pattern
5. End with a practical takeaway

## Tone notes

- Mild humour is fine.
- Strong opinions are fine if backed by experience.
- Don't sound corporate.
- Don't sound like an AI trying to impress.
- If there is a choice, prefer clarity over cleverness.

## AI collaboration note

The final article can be AI-assisted, but it should still feel like Brady's thinking, experience, and lessons.
